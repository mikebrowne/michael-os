# Cursor as the coding executor

MichaelOS needs a coding executor for implementation work without building a full in-repo engineering agent. We chose Cursor as the executor via the official `@cursor/sdk` (`Agent.prompt` with `local: { cwd }` and `settingSources: ["project"]`) so headless runs inherit the repo's `.cursor/rules` (public-safety, secret-handling, thin-slice). Mastra remains the control plane: it generates the spec, authors the acceptance test, runs red/green gates, runs independent preflight, and writes file-based run artifacts under `ai-runs/`. Implementation runs in an isolated git worktree with `npm ci` per run so the operator's main working tree stays untouched. The trade-off is dependency on Cursor (API key, SDK behavior, and local executor availability) plus per-run install latency; we accept this because Cursor is purpose-built for coding and lets MichaelOS focus on spec quality, verification, and observability rather than reimplementing an IDE agent.

Considered alternatives: a custom Mastra engineering agent with file/shell tools (rejected for Phase 2 — reinvents Cursor's coding loop); CLI subprocess to an undocumented binary (rejected — brittle parsing); cloud-only Cursor runs (deferred — local-first Mac mini runtime is the current target).
