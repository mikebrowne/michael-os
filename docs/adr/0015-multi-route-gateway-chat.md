# Multi-route gateway chat and thread model

Phase 3 deferred multi-route gateway chat until the registry pattern was proven ([ADR 0004](./0004-engineering-department-structure.md)). Phase 6 enabled `directChat: true` on the Skill Engineer but left a separate `skill-gateway` script because the main gateway was still hardwired to the Engineering Lead. Phase 4b activates the reserved `directChat` field: the operator may talk to any registered direct-chat agent and switch between them without restarting the gateway.

**Decision:** extend the gateway with a thin **`gatewayRouteRegistry`** that tracks (1) the **active direct-chat agent id** and (2) a **stable Mastra thread id per agent route** (`resource: operator`). The operator switches routes with **`@<agent-id>`**; each route gets its own conversation thread so working memory and message history do not bleed across agents. State persists in gitignored **`.mastra/gateway-routes.json`** (daemon) or in-memory for one-shot CLI sessions. The prompt prefix shows the active agent (`engagement-manager>`, etc.). The default active agent is the **Engagement Manager** once shipped, overridable via `GATEWAY_DEFAULT_AGENT=engineering-lead` for rollback.

This extends [ADR 0006](./0006-gateway-session-memory.md) — same `{ thread, resource }` contract on every `agent.generate()` call — but multiplies threads by route instead of one thread per gateway launch. Cross-session engineering continuity remains on disk (WorkItems, PRDs); memory is still for within-route conversation.

Considered alternatives: **single thread with agent tags in messages** — rejected because thread-scoped working memory would mix EL ship context with Skill Engineer skill edits. **Resource-scoped memory per agent** (`resource: operator:engineering-lead`) — rejected for Phase 4b to avoid Mastra resource proliferation before we need cross-thread operator recall (Phase 8 territory). **Separate daemon per agent** — rejected as operational overhead on the Mac mini. **Keep `skill-gateway` as a second entry point** — rejected; one gateway front door is the north star.

The trade-off is slightly more local state (route map file) and gateway complexity; we accept it because Phase 3 explicitly reserved `directChat` for this cheap extension and the Engagement Manager requires a distinct intake thread from the Engineering Lead's build loop.
