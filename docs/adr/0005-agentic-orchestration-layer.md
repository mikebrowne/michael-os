# Agentic orchestration layer (agent + tools, not workflow)

Phase 2 needed an orchestration layer between the operator and the Cursor coding executor (ADR 0003). We chose **Option 2: agentic** — a single **Engineering Lead** Mastra `Agent` with tools and Matt Pocock–style `SKILL.md` guidance — rather than a Mastra **workflow** that sequences grill → PRD → tests → build → ship. The agent does judgment work (interviewing, drafting, when to call tools); **deterministic code** owns repeatable operations: `runAgentBuild()` enforces build order (RED → Cursor → GREEN + hash lock → preflight), `approvalGate` blocks dangerous tools until the operator replies YES, and `report.ts` derives build status and ship eligibility from structured results — the agent may describe outcomes but cannot override a red build or invent green. Side effects (saving PRDs, GitHub issues, git ship) happen only through Mastra tools invoked from the gateway REPL (`scripts/gateway.ts`). Skills are committed `skills/<name>/SKILL.md` files loaded into agent instructions via `skillLoader.ts`, not the Phase 6 YAML skill platform. The trade-off is less rigid sequencing than a workflow (the agent can drift or skip tool calls) and more operator supervision at checkpoints; we accept this because conversation is the product interface, thin vertical slices are the delivery unit, and code-enforced guards on the genuinely dangerous steps (build, ship) bound the risk while keeping the loop easy to extend.

Considered alternatives: **Mastra workflow for the full engineering loop** — rejected because it couples orchestration to a fixed graph, fights conversational grill and checkpoint pacing, and duplicates judgment the Engineering Lead already performs; the workflow shape is reserved for deterministic sub-pipelines inside `runAgentBuild()` only. **Prompt-only approval for dangerous tools** — rejected; `approvalGate.ts` must intercept `run-build`, `ship-docs`, and `ship-implementation` in code (seed for BL-003 / #1). **LLM-narrated build status** — rejected; D+ reports and ship guards read `RunAgentBuildResult`, not chat text. **Full skill YAML platform now** — deferred to Phase 6 (#6); Phase 2 ships the minimal `SKILL.md` folder convention only.
