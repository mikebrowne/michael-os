# Engineering Department structure: agent registry, role/kind taxonomy, and advisory review gate

Phase 3 grows MichaelOS from a single Engineering Lead into a small **Engineering Department**, and we need a structure that adds agents without rebuilding the gateway each time. We chose a **code-defined agent registry** (`src/mastra/agentRegistry.ts`, type `AgentRegistration[]`) as the single source of truth for the roster, where every entry is a **role (who)** realized by a **kind (how)**: `mastra-agent` (conversational, e.g. Engineering Lead, Code Reviewer), `external-executor` (bounded work, e.g. the Software Engineer role performed by Cursor), or `skill` (judgment loaded onto an agent, e.g. grill/to-prd/tests). The gateway keeps a **single front door** in Phase 3 — the operator only chats with the Engineering Lead, which invokes specialists via handoff — but each registration declares `directChat` and `standalone` so Phase 4 can enable direct operator→agent chat as a config change rather than a refactor. The first specialist hire is a **Code Reviewer** (`mastra-agent` + `code-review` skill) that runs **post-green, pre-ship** and returns a structured verdict (`approve` / `request-changes` / `block`); in Phase 3 its authority is **advisory** — the verdict is surfaced in the report but the operator's explicit YES still ships. Registrations carry an optional `model?` so high-judgment roles (Reviewer) can run a stronger reasoning tier than orchestration roles (Engineering Lead), with the concrete model resolved from config/`.env` rather than hardcoded. The trade-off is that a code-defined registry is less dynamic than a data-driven one and requires a code change to add an agent; we accept this because the roster changes rarely, the registry can hold live agent factory references, and a YAML/data-driven skill+agent platform is explicitly deferred to Phase 6 (#6).

Considered alternatives: a **data-driven registry** (YAML/JSON) — rejected for Phase 3 as premature; it is the Phase 6 skill-platform shape (#6) and cannot hold live factory references. **Multi-route gateway chat now** (operator picks any agent in the REPL) — deferred to Phase 4 because it pulls in session/thread routing and splits operator context before the registry pattern is proven; the reserved `directChat`/`standalone` fields keep it cheap later. **A hard-gating Reviewer** that blocks ship on `block` — rejected for Phase 3 as too much authority for a new agent and as overlapping runtime approval gating (BL-003 / #1); escalation to soft/hard gating is Phase 4/5 work. **A Necessity Reviewer (Ponytail) as the first hire** — deferred to Phase 4 because pre-build necessity review pays off once builds are expensive, whereas the unreviewed green→ship gate is the more urgent risk today. **A dedicated in-harness coding agent** instead of the Cursor-backed Software Engineer role — rejected as a re-litigation of [ADR 0003](./0003-cursor-coding-executor.md). The Phase 2 agentic orchestration shape ([ADR 0005](./0005-agentic-orchestration-layer.md)) and gateway memory model ([ADR 0006](./0006-gateway-session-memory.md)) are assumed, not re-litigated here.
